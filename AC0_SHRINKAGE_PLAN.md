# План следующего этапа (важно сделать сейчас)

Ниже зафиксирован **ровно тот план, который сейчас важно доделать**. Он отражает,
что текущая версия честно использует конструктивный depth‑2 bound `M²`, но для
финального результата (NP ⊄ P/poly без внешних фактов) нужно вернуться к
polylog‑оценке через реальную switching‑лемму. Этот документ фиксирует
последовательность и приоритеты.

---

## 1) Статус `M²`: финал или промежуточный этап?

**Ответ:**
- **Скорее всего, нет**, `M²` не годится как финальная оценка для цели P≠NP.
- **Да**, `M²` — отличный промежуточный, полностью формальный stage для depth‑2.

**Почему (аккуратная формулировка):**
- `AC0SmallEnough : M² ≤ n` ⇒ `M ≤ √n`.
- При типичном `M = n^k` условие малости перестаёт работать (кроме малых `k`).
- Значит, финальные теоремы, вероятно, становятся слабее или не применимы к
  нужным диапазонам параметров (зависит от того, что именно обозначает `M`
  и где требуется `AC0SmallEnough`).

**Вывод:**
- Закрепляем `M²` как **Stage‑1 (axiom‑free depth‑2)**.
- Параллельно держим цель **Stage‑2: вернуть polylog** только через реальную
  switching‑лемму, без bridge‑аксиом.

## Шаг A. Аудит на скрытые аксиомы

Создать единый файл `Audit.lean` и явно проверить аксиомы:


**Цель:** убедиться, что среди внешнего осталось только ожидаемое
(например, `shrinkage_for_localCircuit`), а не «аксиома под видом witness».

Плюс быстро проверить, не стали ли финальные теоремы условными:

```lean

```

**Важно:** `Audit.lean` должен жить в отдельной папке/namespace и не
импортироваться в «боевую» сборку — это аудит, а не часть конвейера.

## Шаг B. Зафиксировать stage‑контракт в интерфейсе

**Вариант 1 (минимальный):**

```lean
def ac0DepthBound (params : AC0Parameters) : Nat := params.M^2
```

И downstream обязан использовать только `ac0DepthBound` как **верхнюю границу**
(то есть использовать неравенства вида `depthBound ≤ ac0DepthBound params`,
а не считать её «точной глубиной»).
Позже это позволит заменить оценку без массового переписывания.

**Вариант 2 (лучше):**
разделить слабый и сильный слои:

- `Facts_Switching_Weak` — конструктивный `M²`
- `Facts_Switching_Strong` — polylog (сейчас можно как цель)

И параметризовать pipeline, какую версию shrinkage используем.
Это даст:
- формально чистый weak‑вариант,
- сильный вариант для end‑to‑end цели,
- возможность постепенно закрывать сильный вариант реальными доказательствами.

**Важно:** `Facts_Switching_Strong` не должен импортироваться в axiom‑free ветку.
Его стоит держать в отдельном namespace/target, чтобы случайно не протащить
сильные допущения в основную сборку.

## Шаг C. Проверить, ломает ли `M²` финальную цель

1) Найти все места использования `AC0SmallEnough` и нового bound:

```bash

```

2) Оценить, превращаются ли ключевые теоремы в условные.
Если да — фиксируем: финальный результат требует polylog.

# Если цель — вернуть polylog (Stage‑2), что формализовать первым

Наиболее практичный и Lean‑дружелюбный маршрут:

1) **Truncation ширины** DNF/CNF до `w`:
   ошибка ≤ `M · 2^{-w}`.
2) **Switching lemma для ширины `w`** (комбинаторное доказательство
   через инъекцию Разборова / конспекты О’Доннелла).
3) **Union bound** на семейство формул (если нужен единый сертификат для `F`).

Это даёт polylog‑bound **без bridge‑аксиом и без тяжёлой вероятности**.
Нам достаточно вероятностного метода через подсчёт конечных множеств
ограничений (например, `3^n`), без measure theory: нужна лишь
**существование** подходящего ограничения, а не конструктивность.
