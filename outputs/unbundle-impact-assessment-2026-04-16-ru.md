# Оценка эффекта unbundle-рефактора (2026-04-16)

## Вопрос

> Изменения с выносом `hMag : MagnificationAssumptions` из сигнатуры `P_ne_NP_final` — это реальный прогресс или ничего не меняет?

## Короткий ответ

Это **реальный инженерный прогресс**, но **не финальный математический прорыв**.

То есть:

1. API/архитектура действительно улучшились;
2. доказательство не стало безусловным;
3. до zero-arg `P_ne_NP` по-прежнему нужен внутренний source для оставшихся
   предпосылок.

## Что поменялось объективно

### 1) Снят bundle-level публичный слой

Раньше default-теорема принимала один большой bundle `hMag`.
Теперь default-API принимает только минимальный явный payload:

- `hMS`,
- `hAsym`,
- `hNPbridge`.

Это снижает «скрытость зависимостей» и делает аудит обязательств заметно проще.

### 2) Сохранена обратная совместимость

Добавлены compatibility-обёртки с `..._of_magnification`, поэтому старые
вызовы не ломаются.

Практически: можно постепенно мигрировать пользователей API без одномоментного
массового refactor-а.

### 3) Улучшена трассируемость следующего математического шага

После unbundle теперь видно точное множество внешних долгов, которые надо
внутренне закрыть, вместо расплывчатого «надо убрать `hMag`».

## Что НЕ поменялось

### 1) Безусловность не достигнута

`P_ne_NP_final` всё ещё не zero-arg theorem: внешние предпосылки остались,
просто теперь они явные.

### 2) Математический blocker не снят

Нужны внутренние доказательства для assumption payload
(`hMS/hAsym/hNPbridge`) или эквивалентный полностью внутренний маршрут.

## Итоговая оценка

- **Инженерная ценность:** высокая (чистота API, совместимость, декомпозиция долга).
- **Математическая ценность до unconditional:** средняя/подготовительная.
- **Статус цели «безусловно»:** не закрыта.

## Почему это всё-таки важно (а не «ничего не меняет»)

Если бы это был «нулевой» change, он бы не уменьшал область неопределённости.
Здесь область неопределённости уменьшена: теперь блокер выражен в виде
трёх конкретных hypothesis-компонент, с которыми можно работать по отдельности.

Это типичный полезный шаг перед сложной internalization-фазой:
сначала распаковка интерфейса, затем закрытие каждого поля доказательствами.
