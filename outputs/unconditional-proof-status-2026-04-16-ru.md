# Текущее состояние доказательства и что осталось до безусловности (снимок на 2026-04-16)

## 0) Короткий вердикт

На текущем `mainline`-маршруте репозиторий **не безусловный** в строгом смысле:
публичная финальная теорема всё ещё зависит от
`hMag : MagnificationAssumptions`.

При этом важная часть уже закрыта:

- DAG-часть separation на дефолтном пути уже внутренне собрана;
- tree `pnp3/` axiom/sorry-clean;
- сборка/аудит проходят.

То есть «остаток до безусловности» сейчас в основном
**formula/magnification internalization**, а не в DAG-проводке.

---

## 1) Что уже доказано/закрыто (по факту в репозитории)

1. **Чистота формализации**: в активном `pnp3/` нет `axiom`, нет `sorry/admit`.
2. **Inclusion-side закрыт**: `P ⊆ PpolyDAG` внутренне доказано и подключено в final-route.
3. **DAG endpoint plumbing закрыт**:
   default DAG-separation получаем внутри, без внешнего DAG-аргумента.
4. **Исторический fixed-slice support-half маршрут закрыт как no-go**
   (не активный путь для unconditional closure).

Практически это означает: инфраструктурный слой уже доведён до состояния,
когда главный долг — это один конкретный публичный контракт (`hMag`).

---

## 2) Главный оставшийся блокер до безусловности

## Blocker A — убрать `hMag : MagnificationAssumptions` из публичной финальной теоремы

### Что именно не закрыто

Сейчас публичный endpoint остаётся вида:

```text
P_ne_NP_final (hMag : MagnificationAssumptions)
```

Пока этот аргумент есть, формально это не zero-argument unconditional theorem.

### Почему это ключевой блокер

Даже при внутренне закрытой DAG-части безусловность в публичном API
ломается на compatibility-пакете magnification assumptions.

### Сложность починки

**Высокая (research-level + API-level)**.

- Нельзя просто «удалить аргумент» синтаксически.
- Нужно либо:
  1) внутренне доказать содержимое `MagnificationAssumptions`,
  2) либо перестроить final theorem так, чтобы это содержимое больше не
     требовалось как вход.

### Идея решения (реалистичный маршрут)

1. Зафиксировать минимальный используемый sub-bundle из `hMag`
   (какие именно поля реально читаются в `NP_not_subset_PpolyFormula_final`
   и downstream-обёртках).
2. По каждому полю:
   - либо дать internal theorem source,
   - либо заменить более слабым, но выводимым локально контрактом.
3. Сделать промежуточный theorem-bridge:
   `NP_not_subset_PpolyFormula_final_internalized` (без внешнего `hMag`).
4. Переподключить `P_ne_NP_final` на этот bridge и свернуть API до zero-arg.

## 3) Вторичные (но важные) долги

## Blocker B — formal source-математика для formula/magnification слоя

### Что именно не закрыто

DAG-часть уже не основной блокер, но formula-side всё ещё зависит
от пакета предположений magnification.

### Сложность починки

**Высокая**.

Это содержательная математика, а не только refactor. Часть доказательств
может потребовать новых промежуточных лемм в `Magnification/*` и/или
`ThirdPartyFacts/*`.

### Идея решения

- Ввести «каркас трассировки зависимостей»:
  для каждого поля `hMag` указать
  `used-by theorem` → `candidate internal source` → `status`.
- Закрывать поля батчами (small batches), не пытаться закрыть всё монолитно.
- После каждого батча прогонять `check.sh` и audit-тесты,
  чтобы не ломать endpoint surface.

---

## Blocker C — согласованное обновление публичной документации после closure

### Что именно не закрыто

Пока `hMag` остаётся, нельзя честно заявлять «безусловно доказано»
в README/STATUS/checklist как финальный факт.

### Сложность починки

**Низкая–средняя** (технически легко, но требует дисциплины синхронизации).

### Идея решения

Сразу после снятия Blocker A:

1. обновить `README.md`, `STATUS.md`, `TODO.md`, `CHECKLIST_*`, `AXIOMS_FINAL_LIST.md`;
2. зафиксировать единый unconditional wording без расхождений между файлами;
3. добавить короткий proof-status changelog entry с датой.

---

## 4) Что НЕ стоит считать активным блокером сейчас

1. **Не DAG plumbing** — он уже закрыт на default path.
2. **Не fixed-slice support-half route** — это исторический no-go branch.
3. **Не отсутствие базовой компиляции** — build/check уже проходят.

Это важно, чтобы не тратить недели на «ремонт» частей,
которые уже не лимитируют выход к безусловности.

---

## 5) Приоритизированный план (конкретно, что делать следующим)

1. **P0 (критично): dependency audit по `hMag`**
   - автоматически/вручную построить список используемых полей
     и мест потребления;
   - отранжировать по «влияние на final endpoint».

2. **P1: internalization batch #1**
   - закрыть самый узкий набор полей,
   - выпустить промежуточный theorem без этих внешних зависимостей.

3. **P2: internalization batch #2..N**
   - повторять до полного снятия внешнего `hMag`.

4. **P3: API cleanup**
   - перевести `P_ne_NP_final` в zero-argument форму,
   - оставить compatibility wrapper только как deprecated bridge (если нужен).

5. **P4: docs + release wording sync**
   - синхронно обновить все статусные документы.

## 6) Оценка трудоёмкости по задачам

- Blocker A (`hMag` elimination): **8/10**
- Blocker B (formula/magnification source math): **9/10**
- Blocker C (docs synchronization): **3/10**

Итоговый practical риск: высокий из-за математического содержания Blocker B,
но архитектурный риск умеренный, потому что DAG-side инфраструктура уже стабилизирована.

---

## 7) Definition of Done для «действительно безусловно»

Считать цель закрытой только когда одновременно выполнено:

1. публичная финальная теорема больше не принимает `hMag`;
2. `P_ne_NP` выводится как zero-argument theorem в активном дереве;
3. `./scripts/check.sh` и audit-набор проходят;
4. документация синхронно переведена на unconditional wording.
